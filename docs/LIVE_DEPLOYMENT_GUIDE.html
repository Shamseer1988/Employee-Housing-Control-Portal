Architecture change — read first

Since this guide was first generated, the in-stack nginx has been removed from docker-compose.prod.yml. TLS termination, the Cloudflare allowlist, and host-based routing now live in a separate "edge proxy" compose stack that shares the external Docker network pug_edge with this one. Every Cloudflare cert / nginx.conf / deploy/ssl/ reference below now applies to that edge proxy stack's folder, not this repo (which no longer has a deploy/ directory).

What still applies verbatim: the Cloudflare dashboard recipe (Origin Cert minting, DNS A record, SSL/TLS mode), the Cloudflare Tunnel / DDNS setup, the office firewall + port-forwarding, and the verification curls.

What changes: cert files go into <edge-proxy-stack>/deploy/ssl/ instead of deploy/ssl/; nginx.conf is edited inside the edge proxy stack; the bring-up sequence requires docker network create pug_edge on the host first.

2. Prerequisites

2.1 Office PC

  • OS: Windows 10/11 Pro (Hyper-V/WSL2 enabled) or Ubuntu 22.04+/Debian 12+ or macOS 13+.
  • CPU: 4 cores minimum (8 recommended for >100 concurrent users).
  • RAM: 8 GB minimum (16 GB comfortable).
  • Storage: 100 GB free (database + backups + uploads + Docker images).
  • Power: UPS strongly recommended — Postgres on a bad shutdown can corrupt the data dir.
  • Network: Wired Ethernet, business-grade ISP if possible. A dynamic IP is fine (we use DDNS in Section 8); a static IP is even better.

4. Cloudflare Origin SSL (15-year cert)

4.1

Set the zone to Full (Strict)

Cloudflare dashboard → parisunitedgroup.com → SSL/TLS → Overview → Full (strict).

Why Full (Strict)

"Flexible" sends plaintext between Cloudflare and your office — defeats the purpose. "Full" doesn't verify the origin cert. "Full (Strict)" forces Cloudflare to validate the Origin Certificate we'll mint below, so a tampered-with origin can't masquerade as yours.

4.2

Mint the Origin Certificate

  1. SSL/TLS → Origin Server → Create Certificate.
  2. Key type: RSA (2048).
  3. Hostnames: accommodation.parisunitedgroup.com (add *.parisunitedgroup.com if you want one cert to cover future subdomains).
  4. Validity: 15 years.
  5. Click Create — two text blocks appear.
4.3

Save the files

On the office PC, in the repo root:

# Paste the "Origin Certificate" block here
notepad deploy\ssl\origin.crt   # Windows
nano    deploy/ssl/origin.crt   # Linux/Mac

# Paste the "Private key" block here
notepad deploy\ssl\origin.key
nano    deploy/ssl/origin.key
Windows users — line endings

VS Code / Notepad on Windows may save with CRLF. In VS Code: bottom-right status bar → click CRLF → choose LF → save. Nginx accepts both, but LF avoids future surprises.

Never commit these files

.gitignore already excludes deploy/ssl/* except the README and .example stubs. If you accidentally git add origin.key: revoke the cert in Cloudflare and mint a new one. A leaked private key is leaked forever.

4.4

File permissions

# Linux/Mac
chmod 644 deploy/ssl/origin.crt
chmod 600 deploy/ssl/origin.key

# Windows: right-click origin.key → Properties → Security →
# restrict to Administrators + your user. ACLs only — the container
# mounts read-only either way.

5. Production environment file

5.1

Create .env at repo root

The prod compose reads from .env in the same directory as docker-compose.prod.yml. Create it:

# Linux/Mac
cp backend/.env.example .env

# Windows
copy backend\.env.example .env
5.2

Generate strong secrets

# Run twice — once for SECRET_KEY, once for JWT_SECRET_KEY
python -c "import secrets; print(secrets.token_urlsafe(48))"

Or on any Linux/Mac/WSL2 shell:

openssl rand -base64 48
5.3

Edit the values

Open .env and set the following at minimum:

FLASK_ENV=production

# Strong secrets from step 5.2 (≥32 chars, never reused elsewhere)
SECRET_KEY=<paste-token-1>
JWT_SECRET_KEY=<paste-token-2>

# Postgres — pick a strong password
POSTGRES_DB=pug_accommodation
POSTGRES_USER=pug
POSTGRES_PASSWORD=<strong-random-password>

# Cookies are HTTPS-only in prod
JWT_COOKIE_SECURE=true
JWT_COOKIE_SAMESITE=Lax

# CORS — list the actual frontend origin. NEVER '*' here; prod validation rejects it.
CORS_ORIGINS=https://accommodation.parisunitedgroup.com

# Initial admin (you'll change the password on first login)
SUPERUSER_USERNAME=admin
SUPERUSER_EMAIL=admin@parisunitedgroup.com
SUPERUSER_PASSWORD=<temporary-strong-password>

# Tuning
GUNICORN_WORKERS=4
Production validation is strict

Backend boot will refuse to start if any of these are missing, still set to the dev default, shorter than 32 chars, or if CORS_ORIGINS contains *. This is intentional — the safety net stops a misconfigured deploy from ever serving traffic.

6. Office network — port forwarding & firewall

6.1

Find the office PC's LAN IP

# Windows
ipconfig | findstr IPv4

# Linux
ip -4 addr show | grep inet

Note it — e.g. 192.168.1.50. Reserve it in your router's DHCP table so it never changes.

6.2

Open the Windows / OS firewall

# Windows (PowerShell as Administrator)
New-NetFirewallRule -DisplayName "HTTP-in"  -Direction Inbound -LocalPort 80  -Protocol TCP -Action Allow
New-NetFirewallRule -DisplayName "HTTPS-in" -Direction Inbound -LocalPort 443 -Protocol TCP -Action Allow

# Linux (ufw)
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
6.3

Router port-forwarding

On the office router admin page (typically 192.168.1.1):

External portInternal IPInternal portProtocol
80192.168.1.50 (office PC)80TCP
443192.168.1.50 (office PC)443TCP
CGNAT check

If your ISP gives you a 100.64.x.x address on the WAN, you're behind Carrier-Grade NAT — port forwarding won't reach you from the public Internet. Call the ISP and ask for a public IP (sometimes free for business plans), or use Cloudflare Tunnel as an alternative — see Troubleshooting §12.

7. Cloudflare DNS A record

7.1

Find your public IP

curl -s https://ifconfig.me
# or visit  https://whatismyipaddress.com  from the office network
7.2

Add the record

Cloudflare → parisunitedgroup.com → DNS → Records → Add record:

TypeA
Nameaccommodation
IPv4 addressyour public IP from 7.1
Proxy statusProxied (orange cloud)
TTLAuto
Why "Proxied"

The orange cloud is the entire point: Cloudflare's edge terminates the TLS the browser sees, runs WAF/DDoS in front of you, and hides your office IP from the public. The nginx allowlist in deploy/nginx.conf assumes traffic is coming through Cloudflare; bypass attempts get a 403.

8. Dynamic IP auto-updater (Cloudflare DDNS)

If your ISP gives you a dynamic IP (typical for office DSL/fibre without a static-IP add-on), the A record from §7 will go stale the moment your ISP renews the lease. We solve this by running a tiny container that watches your public IP and updates the Cloudflare A record automatically.

Recommended

Use favonia/cloudflare-ddns — a 9 MB, single-purpose container with no external deps, native Cloudflare API, IPv4 + IPv6 support, runs as non-root. Officially listed in Cloudflare's DDNS partners.

8.1

Create a scoped Cloudflare API token

  1. Cloudflare → top-right avatar → My Profile → API Tokens → Create Token.
  2. Template: Edit zone DNS.
  3. Permissions: Zone → DNS → Edit.
  4. Zone Resources: Include → Specific zone → parisunitedgroup.com.
  5. (Optional but recommended) Client IP Address Filtering: Is in → your office IP range.
  6. Continue to summary → Create Token.
  7. Copy the token once — it's never shown again.
8.2

Add the DDNS service to docker-compose.prod.yml

Append this block at the bottom of the services: section (before the volumes: block):

  ddns:
    image: favonia/cloudflare-ddns:latest
    restart: unless-stopped
    network_mode: host        # needed so it sees the host's public IP
    user: "1000:1000"
    read_only: true
    cap_drop: [all]
    security_opt: [no-new-privileges:true]
    environment:
      CLOUDFLARE_API_TOKEN: ${CLOUDFLARE_API_TOKEN:?set CLOUDFLARE_API_TOKEN}
      DOMAINS: accommodation.parisunitedgroup.com
      PROXIED: "true"
      IP4_PROVIDER: cloudflare.trace
      IP6_PROVIDER: none      # flip to cloudflare.trace if you have native IPv6
      UPDATE_CRON: "@every 5m"
      TZ: Asia/Riyadh
8.3

Drop the token into .env

CLOUDFLARE_API_TOKEN=<the-token-from-8.1>
8.4

Launch + verify

docker compose -f docker-compose.prod.yml up -d ddns
docker compose -f docker-compose.prod.yml logs -f ddns

You should see a log line like:

Updating A record of accommodation.parisunitedgroup.com (was 1.2.3.4, now 5.6.7.8)

From then on, the container re-checks every 5 minutes and updates the A record only when the IP actually changes.

Alternative: Cloudflare Tunnel

If your ISP blocks inbound 80/443 or puts you behind CGNAT, replace the DDNS approach entirely with Cloudflare Tunnel — Cloudflare establishes an outbound connection from your office to the edge, and you don't need port forwarding or a public IP at all. The tradeoff is one extra moving piece. See Cloudflare Tunnel docs.

9. First deployment

9.1

Verify Cloudflare IP ranges

Open deploy/nginx.conf and compare the set_real_ip_from + geo $cf_edge blocks against cloudflare.com/ips-v4 and /ips-v6. If anything changed, update both blocks. See Appendix A for the procedure.

9.2

Build and boot

docker compose -f docker-compose.prod.yml build --no-cache
docker compose -f docker-compose.prod.yml up -d

First build takes ~5–10 minutes. The backend boot chain (wait-for-db → init-db → migrate-all → seed → gunicorn) takes another ~30 seconds.

9.3

Tail the logs

docker compose -f docker-compose.prod.yml logs -f backend nginx

Expected:

  • backend: Database reachable (attempt 1) → seed messages → Starting gunicorn with Using worker: gthread.
  • nginx: no errors; first GET /health at startup.

git pull

docker compose -f docker-compose.prod.yml up -d

Zero-downtime for read traffic; a 5–10 second blip during the backend swap.

11.2 Backups

  • Automatic: Settings → Backup → schedule (daily by default at 03:00 UTC). Retention configurable.
  • Manual: Settings → Backup → Backup now.
  • Off-host copy: point Settings → Backup → "Backup folder path" at /mnt/host-backups and add a bind mount in compose to a network share / OneDrive / external drive.
  • Verify weekly: Download the latest .dump, restore into a throwaway container, confirm it loads.

11.3 Logs

# Live tail of one service
docker compose -f docker-compose.prod.yml logs -f --tail=200 backend

# All services, last 5 min
docker compose -f docker-compose.prod.yml logs --since=5m

11.4 Restart a single service

docker compose -f docker-compose.prod.yml restart backend

11.5 Full restart (after host reboot)

docker compose -f docker-compose.prod.yml up -d

Compose will restart every service in dependency order. restart: unless-stopped means Docker also brings everything back automatically when the PC reboots, as long as Docker Desktop is set to start with Windows (Settings → General → "Start Docker Desktop when you sign in to your computer").

12. Troubleshooting

12.1 Browser shows a Cloudflare error page

CodeLikely causeFix
522Origin unreachableOffice PC off, router port forward broken, ISP blocks 443 inbound, or public IP changed (DDNS not running).
525SSL handshake failedOrigin cert files missing/corrupt, wrong line endings, or zone is in "Full (Strict)" but origin cert doesn't cover the hostname.
526Invalid SSL certificateYou're on "Full (Strict)" with a self-signed cert. Use Cloudflare Origin Certificate (Section 4).
520Empty response from originnginx not running, or the request crashed it. docker compose -f docker-compose.prod.yml logs nginx.

12.2 403 Forbidden from nginx

Means your nginx geo $cf_edge allowlist rejected the request. Either:

  • The Cloudflare IP ranges in deploy/nginx.conf are stale → run the update procedure in Appendix A.
  • You're hitting the public IP directly without Cloudflare in front → use the hostname.

12.3 Backend won't boot — "Insecure production configuration"

You missed a value in .env. The error lists every problem; fix them all and re-run docker compose -f docker-compose.prod.yml up -d backend.

12.4 Login "succeeds" but bounces to /login

Almost always cookies being dropped. Confirm:

  • You're hitting the site over HTTPS, not http://.
  • JWT_COOKIE_SECURE=true in .env (already the default in the prod compose).
  • Cloudflare zone is on Full (Strict), not Flexible.

12.5 DDNS isn't updating

docker compose -f docker-compose.prod.yml logs ddns | tail -50

Common causes:

  • Token doesn't have Zone:DNS:Edit on the right zone (re-mint from 8.1).
  • Token IP filter doesn't include your current office IP.
  • The A record doesn't exist yet — create it manually once (Section 7).

12.6 ISP blocks inbound 80/443 (CGNAT)

Switch to Cloudflare Tunnel:

# Replace the public DNS / port-forward setup. Install cloudflared,
# auth, route the tunnel to http://localhost:80, point the DNS at the tunnel.
# Full guide: https://developers.cloudflare.com/cloudflare-one/connections/connect-networks/get-started/

Nothing in the application needs to change — Tunnel just replaces the router/DDNS layer.

Appendix A — Updating Cloudflare IP ranges

Cloudflare occasionally adds new edge subnets. Stale ranges in deploy/nginx.conf have two failure modes: (a) real Cloudflare requests get 403'd from a new edge IP, or (b) the set_real_ip_from trust list is too small and an attacker spoofs CF-Connecting-IP from a different range.

A.1

Pull the current lists

curl -s https://www.cloudflare.com/ips-v4 -o cf-v4.txt
curl -s https://www.cloudflare.com/ips-v6 -o cf-v6.txt
A.2

Diff against the repo

# Linux/Mac
diff <(grep -oE '[0-9.]+/[0-9]+' deploy/nginx.conf | sort -u) <(sort -u cf-v4.txt)
A.3

Update both blocks

If the diff is non-empty, edit deploy/nginx.conf and update:

  • The set_real_ip_from <range>; lines near the top.
  • The geo $cf_edge { ... } block below them.

Then rebuild nginx:

docker compose -f docker-compose.prod.yml build nginx
docker compose -f docker-compose.prod.yml up -d nginx
Set a calendar reminder

Cloudflare publishes updates infrequently but without notice. A quarterly reminder ("verify Cloudflare IPs") is plenty.
