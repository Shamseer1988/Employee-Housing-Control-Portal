PUG Accommodation Portal — Setup Guide End-to-end instructions: from git clone to logging into the running portal at http://localhost:3000.

Two ways to run this on Windows

Option A — Docker Desktop (fastest path). One install, one command, the same images you'd run in production. Skip straight to section 0: Docker Compose. ~5 minutes to first login.

Option B — Native install. Best for active coding (Flask + Next dev servers auto-reload on save). Follow sections 01–10 below.

00 Docker Compose path (recommended for demos)

If all you want is to run the portal — for a demo, QA pass, or to evaluate it before deciding on production deployment — this is the shortest path. Skip every section after this one; you don't need Python, Node, or PostgreSQL on your Windows host.

0.1 Install Docker Desktop

ToolVersionDownloadDuring install
Git for Windows 2.40+ git-scm.com Default options.
Docker Desktop 4.x docker.com Leave "Use WSL 2 instead of Hyper-V" ticked (default). Reboot when prompted.

After installing, open Docker Desktop once and wait for the whale icon (bottom-left) to turn green. Verify in PowerShell:

PowerShellPS> docker --version
PS> docker compose version
Windows prerequisites Docker Desktop checks for you
  • Windows 10 22H2 / Windows 11 (64-bit).
  • CPU virtualization enabled in BIOS (the installer warns if not).
  • WSL 2 (the installer offers to install it if it's missing).

If port 80 is taken on Windows IIS, an older Skype install, or another web server may already be bound to port 80. Change two lines in .env: 
HTTP_PORT=8080
PUBLIC_BASE_URL=http://localhost:8080
CORS_ORIGINS=http://localhost:8080
Then open http://localhost:8080 instead.

0.4 Build the images

PowerShellPS> docker compose build

First build is ~3–5 minutes (downloads the base images, installs pip + npm packages, runs next build). Subsequent builds are much faster thanks to layer caching.

0.5 Start everything

PowerShellPS> docker compose up -d

This starts four containers in dependency order: db → backend → frontend → nginx. The backend container automatically runs flask db upgrade (creates tables), then flask seed (36 permissions, 9 roles, 44 settings, admin user), then starts Gunicorn.

Watch the boot live in a second PowerShell window:

PowerShellPS> docker compose logs -f backend

You'll see INFO [alembic.runtime.migration] Will assume transactional DDL, the seed lines, then [gunicorn] Listening at: http://0.0.0.0:5000. Press Ctrl + C to stop tailing — the containers keep running.

0.7 Docker cheat-sheet

PowerShell# Tail logs from any service
PS> docker compose logs -f backend
PS> docker compose logs -f frontend
PS> docker compose logs -f nginx

# Open an interactive Flask shell inside the backend
PS> docker compose exec backend flask --app wsgi shell

# Re-run the seed (idempotent — refreshes permissions/settings)
PS> docker compose exec backend flask --app wsgi seed

# Connect to Postgres directly
PS> docker compose exec db psql -U pug -d pug_accommodation

# Run the test suite inside the running container
PS> docker compose exec backend pytest -q --disable-warnings

# Stop everything (keeps the DB and uploads)
PS> docker compose down

# Start again
PS> docker compose up -d

# Restart a single service
PS> docker compose restart backend

# Rebuild after pulling new code
PS> git pull
PS> docker compose build
PS> docker compose up -d

# Nuke EVERYTHING including the database (dev only — irreversible)
PS> docker compose down -v

0.8 Backup & restore on Windows

The shell scripts in scripts/ are bash, so on Windows talk to the container directly with PowerShell:

PowerShell — backupPS> mkdir backups -Force
PS> $stamp = Get-Date -Format "yyyyMMddTHHmmssZ"
PS> docker compose exec -T db pg_dump -U pug pug_accommodation | Out-File -FilePath "backups/pug-$stamp.sql" -Encoding utf8
PowerShell — restorePS> docker compose exec -T db psql -U pug -d pug_accommodation -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"
PS> Get-Content backups/pug-20260101T000000Z.sql | docker compose exec -T db psql -U pug -d pug_accommodation
PS> docker compose restart backend

0.10 Docker troubleshooting

SymptomFix
error during connect: ... open //./pipe/dockerDesktopLinuxEngine Docker Desktop isn't running. Launch it from the Start menu and wait for the green whale icon.
Ports are not available: ... bind: address already in use Port 80 is taken on your machine. Change HTTP_PORT, PUBLIC_BASE_URL, and CORS_ORIGINS in .env as shown in §0.3, then docker compose up -d again.
WSL 2 installation is incomplete dialog on first run Run wsl --install in an Administrator PowerShell, reboot, relaunch Docker Desktop.
Frontend container exits with build errors mentioning NEXT_PUBLIC_API_URL Make sure PUBLIC_BASE_URL in .env matches what the browser actually uses. Rebuild: docker compose build frontend && docker compose up -d.
Login returns "Invalid credentials" docker compose exec backend flask --app wsgi seed to recreate the admin user, then try the password from SUPERUSER_PASSWORD in .env.
Want to start over from a clean database docker compose down -v deletes the named volumes (including pug_pgdata), then docker compose up -d recreates everything from scratch.
Docker Desktop using too much RAM Open Docker Desktop → Settings → Resources → cap WSL Memory to e.g. 4 GB.

01 Prerequisites & installers

Install these four tools first. Use the official Windows installers; default options are fine unless noted.

ToolVersionDownloadDuring install
Git for Windows 2.40+ git-scm.com Pick "Git from the command line and 3rd-party software". Default everything else.
Python 3.11.x or 3.12.x python.org Tick "Add python.exe to PATH" on the first installer screen.
Node.js 20.x LTS nodejs.org Default. The installer adds Node and npm to PATH automatically.
PostgreSQL 14.x – 16.x postgresql.org Remember the postgres superuser password you set. Leave port at 5432. Tick "Stack Builder" only if you want extras (you don't need it).
PowerShell execution policy If activating the Python virtual environment later complains about scripts being disabled, run this once in PowerShell (as your normal user): 
PowerShellPS> Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
Answer Y. This is safe — it allows locally-created and signed scripts.
Recommended terminal Use Windows Terminal with PowerShell 7 (install from Microsoft Store). Every command below assumes PowerShell unless labelled CMD or psql.

02 Verify every tool is installed

Open a new PowerShell window and confirm each one prints a version. Close and reopen any old terminals after installing so PATH updates are picked up.

PowerShellPS> git --version
PS> python --version
PS> pip --version
PS> node --version
PS> npm --version
PS> psql --version
If psql is not recognised PostgreSQL adds its bin folder to PATH only when the installer detects an existing PATH variable. Manually add it:
  1. Press Win + R → type sysdm.cpl → Enter.
  2. Tab "Advanced" → Environment Variables → under "User variables" select Path → Edit → New.
  3. Add C:\Program Files\PostgreSQL\16\bin (adjust version number).
  4. Click OK three times. Open a fresh PowerShell and re-run psql --version.

04 Create the PostgreSQL database & user

Open the psql shell as the superuser. You'll be prompted for the password you set during PostgreSQL installation.

PowerShellPS> psql -U postgres

The prompt changes to postgres=#. Paste the following four lines exactly:

psqlCREATE USER pug WITH PASSWORD 'pug_local_pass';
CREATE DATABASE pug_accommodation OWNER pug;
GRANT ALL PRIVILEGES ON DATABASE pug_accommodation TO pug;
\q

You're back at PowerShell. Confirm the database exists and your new user can log in:

PowerShellPS> psql -U pug -d pug_accommodation -c "SELECT current_user, current_database();"

Enter the password pug_local_pass when prompted. You should see:

 current_user |  current_database
--------------+---------------------
 pug          | pug_accommodation
(1 row)
Local password ≠ production The pug_local_pass password is for local development only. Pick a strong one for any shared environment and never commit it.

05 Backend setup (Flask)

5.1 Create the Python virtual environment

PowerShellPS> cd C:\dev\Employee-Housing-Control-Portal\backend
PS> python -m venv .venv
PS> .\.venv\Scripts\Activate.ps1

Your prompt now starts with (.venv). If you ever close the terminal, just re-run that Activate.ps1 line to re-enter the venv.

CMD instead of PowerShell? Activate with .venv\Scripts\activate.bat instead.

5.2 Install dependencies

PowerShell (venv)(.venv) PS> python -m pip install --upgrade pip
(.venv) PS> pip install -r requirements.txt

This takes ~30 seconds. It installs Flask, SQLAlchemy, Flask-Migrate, psycopg2, openpyxl, etc.

5.3 Configure environment variables

PowerShell (venv)(.venv) PS> Copy-Item .env.example .env
(.venv) PS> notepad .env

Notepad opens. Replace the file with these values (keep your own secrets if you want):

.envFLASK_APP=wsgi.py
FLASK_ENV=development
FLASK_DEBUG=1
SECRET_KEY=local-dev-secret-change-me
JWT_SECRET_KEY=local-dev-jwt-secret-min-32-bytes-long-12345

DATABASE_URL=postgresql+psycopg2://pug:pug_local_pass@localhost:5432/pug_accommodation

CORS_ORIGINS=http://localhost:3000

UPLOAD_FOLDER=../uploads
MAX_CONTENT_LENGTH_MB=25

# Default super user (created on first `flask seed` run)
SUPERUSER_USERNAME=admin
SUPERUSER_EMAIL=admin@pugroup.local
SUPERUSER_PASSWORD=ChangeMe123!

Save (Ctrl + S) and close Notepad.

5.4 Initialise migrations and create tables

PowerShell (venv)(.venv) PS> flask --app wsgi db init
(.venv) PS> flask --app wsgi db migrate -m "initial"
(.venv) PS> flask --app wsgi db upgrade
What just happened
  • db init creates a fresh backend/migrations/ folder. Run it only once.
  • db migrate auto-detects every SQLAlchemy model in the app and produces a migration script.
  • db upgrade applies the script — your pug_accommodation database now has all the tables.
Future model changes follow the same pattern: db migrate -m "describe change" → db upgrade.

5.5 Seed permissions, roles & the super user

PowerShell (venv)(.venv) PS> flask --app wsgi seed

Expected output:

Seeding permissions...
  → 36 permissions present
Seeding roles...
  → 9 roles present
Seeding super user...
  → created super user 'admin' (password from SUPERUSER_PASSWORD)
Seeding system settings...
Done.

5.6 Start the API

PowerShell (venv)(.venv) PS> python wsgi.py

The Flask dev server prints * Running on http://0.0.0.0:5000. Leave this terminal open. Quick smoke test in a second PowerShell window:

PowerShell (new window)PS> curl http://localhost:5000/api/v1/health
Expected A JSON response like { "success": true, "data": { "status": "healthy", ... } }.

07 First login & smoke test

  1. Open http://localhost:3000 in your browser.
  2. You'll be redirected to /login. Sign in with:
    • Username: admin
    • Password: ChangeMe123!
  3. You land on the Dashboard. Empty for now — that's expected.
  4. Walk through a quick happy path to confirm the stack works:
  • Settings → toggle the UI / Theme accent color → header & buttons reflect the change.
  • Divisions → "New division" → save → row appears with auto-code DIV-0001.
  • Landlords → "New landlord" → save → LL-0001.
  • Properties → "New property" → PROP-0001.
  • Open the property → Floors tab → add a floor → Rooms tab → add a room → add a bed (PROP-0001-F1-R101-B1).
  • Employees → "New employee" → save → EMP-00001.
  • Transactions → New assignment → wizard → post → bed flips to occupied.
Change the admin password Before anyone else uses the instance, sign in as admin, hit your name in the top right, and update the password. The seed CLI is idempotent but it never overwrites a manually-changed password.

10 Troubleshooting

SymptomFix
Activate.ps1 cannot be loaded because running scripts is disabled Run Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser once, then re-activate.
psql: command not found after PostgreSQL install Add C:\Program Files\PostgreSQL\<ver>\bin to User PATH (see §02). Open a fresh terminal.
pip install fails on psycopg2-binary compile Confirm Python is 3.11 or 3.12 (3.13 wheels weren't ready at the time of writing). python --version.
FATAL: password authentication failed for user "pug" The password in backend/.env doesn't match the one you set during CREATE USER. Re-open backend/.env and align them.
FATAL: database "pug_accommodation" does not exist Re-run step 04 — the CREATE DATABASE line was skipped or used a different name.
flask: command not found The virtual env isn't active. Re-run .\.venv\Scripts\Activate.ps1 from inside backend\.
Target database is not up to date on db migrate Run flask --app wsgi db upgrade first, then db migrate.
EADDRINUSE :3000 or :5000 Another process is on that port. Find it: netstat -ano | findstr :5000 → note the PID → taskkill /F /PID <pid>.
Frontend loads but every API call shows CORS error in DevTools CORS_ORIGINS in backend/.env must include the exact origin (http://localhost:3000, no trailing slash). Restart the backend after editing.
Login returns "Invalid credentials" with the seeded admin Re-run flask --app wsgi seed. If you'd already changed the password in the UI, use that one instead.
Want to start over from a clean database 
psql -U postgres -c "DROP DATABASE pug_accommodation;"
psql -U postgres -c "CREATE DATABASE pug_accommodation OWNER pug;"
Then re-run flask db upgrade and flask seed.
